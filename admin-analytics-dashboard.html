Implementation Notes — Admin Analytics Dashboard v3

Verified against live schema (evil-panther, 2026-06-26). New in v3: the date range now actually drives every range-bound metric & chart (snapshot cards stay fixed by design), date-based chart axes, chart loading + empty states, keyboard-accessible tooltips, undo + confirm on moderation actions, and responsive breakpoints. The prototype's range data is generated client-side for demo purposes — production reads precomputed snapshots.

Sidebar — accordion nav

  • Analytics is one collapsible section within the broader admin panel nav — not a standalone sidebar. Other sections (Members, Content, Monetization, Settings, Moderation) are peer accordion groups.
  • The accordion state should persist across page navigations (store in localStorage or URL param).
  • Active section badge (open reports count) should come from a lightweight polling endpoint or SSE — not a full page reload.
  • On mobile, the sidebar should collapse to a drawer triggered by a hamburger — not shown in this prototype.

Date range picker

  • Predefined ranges are computed at query time server-side, not stored. The frontend sends from + to ISO dates.
  • "This week" = Monday of current ISO week → today. "Last week" = prior Monday → Sunday. Use the community's configured timezone, not the browser's.
  • Quarter boundaries: Q1=Jan–Mar, Q2=Apr–Jun, Q3=Jul–Sep, Q4=Oct–Dec. Today (Jun 26) = Q2.
  • Half-year: H1=Jan–Jun, H2=Jul–Dec. "Last half-year" from Jun 26 = H2 2025 (Jul 1 – Dec 31, 2025).
  • Custom range: validate from <= to and cap range at 365 days to avoid slow queries.
  • All trend charts re-render on range change — debounce 300ms before firing the API call.
  • Not every metric is range-bound. Cards tagged now (Total Members, Active Subs, MRR, Open Reports) are current snapshots; all-time (Suspended Members) ignores the range entirely. Untagged cards = filtered by the selected range. Keep the .sc-scope chip in sync with the query so the label never lies.

Revenue — Plans + Products

  • Both revenue types use the order collection. Split by order.type: subscription = plan revenue, product = one-time product revenue.
  • MRR is subscription-only. One-time product revenue is non-recurring and shown separately.
  • A community can have both plans and products active simultaneously — the revenue donut shows both segments always.
  • Currency normalization: query order.currency per row; apply an exchange rate snapshot for USD equivalence. Store rates nightly in a fx_rates table (not yet in schema).
  • Refund amount on order.refund_amount is partial or full — always subtract from gross revenue for net figures.

Info tooltips

  • Tooltips use CSS-only positioning (position:absolute on the info button). No JS required for basic hover.
  • Cards near the right edge of the viewport may clip tooltips — flip the tooltip left using a .tip-left modifier when the button is within 220px of the viewport right edge.
  • On touch devices, replace hover with a tap-to-show behaviour (toggle a class). Tooltips shouldn't appear on tap-and-hold.
  • Tooltip copy is the implementation detail, not a help text for members — keep it admin-facing and technical.

Data sources — key fields

  • Members: directus_users.status, last_access, onboarding_completed_at, provider
  • Posts: post.date_created, post_type, reactions_count (precomputed), comments_count (precomputed)
  • Revenue: order.amount, order.type, order.status, order.currency, order.refund_amount
  • Subscriptions: subscription.status, billing_cycle, join to plan for pricing
  • Courses: course_enrollment.completion_percentage, access_status, cert_issued_at
  • Moderation: content_report (needs status field), post_comment.is_rejected, directus_users.status = suspended

Performance

  • All time-series charts need GROUP BY day — precompute daily snapshots in core-service into a community_analytics_snapshot collection. Do not run live aggregates per chart.
  • Stat card counts (members, active subs, reports) can be cached at 5-min TTL — they don't need to be real-time.
  • Activity feed = action_log ORDER BY date_created DESC LIMIT 20 — this is a live query, add a DB index on date_created if not already present.
  • Revenue is admin/owner only — never expose to moderator roles. Gate at the extension hook layer.
  • Export CSV: generate server-side from the same scoped + range-filtered query. Stream the response; do not buffer the entire result set in memory.
